20210707——Redis持久化（八）

本文最后更新于：几秒前

Redis 是内存数据库，如果不将内存中的数据落入磁盘，那么服务器一旦异常崩溃，数据库中的状态也会消失。

RDB redis database


在指定的时间间隔内将内存中的数据集快照写入磁盘，也就是Snapshot快照，他恢复时将快照文件直接读到内存中。

Redis会创建（fork）一个子进程来进行持久化，会将数据临时写入一个文件，待持久化过程结束了，在用这个临时文件替换上次持久好的文件，整个过程中，主进程是不进行任何IO操作的，这就确保了极高的性能，如果需要大规模的数据恢复，且对于数据恢复的完整性不是非常敏感，rdb方式比aof方式更加高效，rdb的缺点是持久化最后一次的数据可能丢失。默认是rdb，一般情况下不需要修改配置。

rdb保存的文件，dump.rdb

触发机制

save规则满足的情况下，会自动触发规则
执行flushall命令，也会触发我们的rdb的规则
退出redis，也会产生rdb文件

备份会生成一个rdb文件

恢复rdb文件

只需要rdb文件，放在redis的启动目录下，
redis启动的时候会自动检查文件，恢复数据

优点：适合大规模的数据，如果对数据完整性要求不高
缺点：需要设置一定的时间间隔进行修改，如果redis意外宕机，那么最后一次修改的数据就没了。fork进程的时候会占据一些内存空间。

AOF append only file

将我们的所有命令都记录下来，如果恢复的时候，就按照这个文件在恢复一遍


以日志的形式来记录每个操作，将Redis执行过的所有指令都记录下来（读操作不记录），只许追加文件但不可以改写文件，redis启动之初会读取文件重新构建数据，换言之，redis重启的话就根据日志文件的内容将指令从前到后执行一次，以完成数据的恢复工作

默认不开启，手动配置，开启yes

超过64mb，太大了，fork一个子进程对文件进行重写

如果aof文件有错位，这时候redis是启动不起来的，我们需要修复配置文件aof
redis给我提供了工具，redis-check-aof –fix

优点：每一次修改都同步，文件的完整会更好，每秒同步一次，可能会丢失一秒的数据，从不同步，效率最高

缺点：相对于数据文件来说，aof远远大于rdb，修复的速度也慢。aof运行效率比rdb慢，我们默认的配置就是rdb，而不是aof
